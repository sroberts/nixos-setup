# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A NixOS flake configuration for a single host: a Framework 13 AMD (Ryzen 7040) laptop running niri + DankMaterialShell on an encrypted LVM-on-LUKS disk with hibernation-capable encrypted swap. It is the **source of truth** for the system — the running machine is a function of these files.

The repo is built to be cloned onto a fresh machine from the NixOS live ISO and installed against. The host-specific `hardware-configuration.nix` is intentionally **not** in the repo (gitignored) — it's generated per-machine by `nixos-generate-config` and dropped in alongside the flake at install time.

## Commands

All builds go through the flake. The host attribute is `framework13`.

```bash
# Rebuild and switch (the default verb after any edit)
sudo nixos-rebuild switch --flake .#framework13

# Test a change without making it the default boot generation
sudo nixos-rebuild test --flake .#framework13

# Build only, don't activate
sudo nixos-rebuild build --flake .#framework13

# Bump nixpkgs / niri / dms / claude-code-nix
nix flake update
# Or one input at a time (Nix 2.19+ positional arg; `--update-input` is deprecated)
nix flake update dms

# Roll back the last activation (pair with `git revert` to keep repo + system aligned)
sudo nixos-rebuild --rollback switch

# Check the flake evaluates and outputs are well-formed
nix flake check
```

There are no tests, no linter, no build matrix — `nix flake check` + a successful `nixos-rebuild` is the whole CI surface.

## Architecture

Three Nix files, one role each:

- **`flake.nix`** — Inputs (nixpkgs unstable, home-manager, nixos-hardware, niri-flake, DankMaterialShell, claude-code-nix) and the single `nixosConfigurations.framework13` output. All inputs `follows = "nixpkgs"` so there is one nixpkgs in the closure. The `lanzaboote` input is **commented out** by design — see *Secure Boot* below.
- **`configuration.nix`** — System-level: bootloader, LUKS resume target, PipeWire, networking, niri + dms-greeter, Docker, Ollama (ROCm), the `scott` user, and the system-wide GUI apps (`_1password-gui`, `chromium`, `obsidian`, …). The function signature is `{ config, pkgs, inputs, … }`. When enabling Secure Boot, **`lib` must be added** to that signature.
- **`home.nix`** — User-level (home-manager): DankMaterialShell config, niri input settings, CLI/TUI tooling, zsh + integrations (zoxide, fzf, eza, bat, starship, mise), and four `home.activation.*` hooks for the imperative gaps that Nix can't declare (pipx tools, LazyVim starter, CyberChef download, post-install TODO.md).
- **`hardware-configuration.nix`** — *Not in the repo.* Generated by `nixos-generate-config` on the target machine; encodes the LUKS UUID and filesystem layout. Lives only at `/etc/nixos/hardware-configuration.nix` (which is symlinked from this working copy after install — see `INSTALL.md`).

### Disk layout — two supported paths

**Default (Calamares install):** ESP + LUKS-encrypted ext4 root, no LVM, no dedicated hibernation swap. Encryption works; hibernation does not. `boot.resumeDevice` is **commented out** in `configuration.nix` for this path. Hardware UUIDs come from `hardware-configuration.nix` (auto-generated by Calamares).

**Appendix (manual LVM-on-LUKS, for hibernation):**

```
nvme0n1p1  ESP (FAT32, unencrypted)
nvme0n1p2  LUKS2 → LVM "vg"
             vg/swap  92 GiB  (encrypted, holds hibernation image)
             vg/root  rest    (encrypted, ext4)
```

This requires uncommenting `boot.resumeDevice = "/dev/vg/swap"` in `configuration.nix` before the install. The path is a stable LVM device, independent of `hardware-configuration.nix`. Hibernation needs persistent-key encrypted swap ≥ RAM, which is why swap lives *inside* LUKS rather than as a random-key swap partition.

The Calamares teardown lines in `configuration.nix` (`services.xserver.enable = false`, `services.displayManager.gdm.enable = false`, `services.desktopManager.gnome.enable = false`) are harmless on the manual path — they're disabling things that were never installed.

### niri + DMS wiring

System side enables `programs.niri` and `services.displayManager.dms-greeter`. User side imports `inputs.dms.homeModules.{dank-material-shell,niri}` and enables `programs.dank-material-shell` with `niri.enableKeybinds = true` and `niri.enableSpawn = true` — those two flags are how DMS auto-starts with niri and registers its keybinds. The greeter reads `~/.config/DankMaterialShell/settings.json` so its theme tracks the user shell's theme.

`systemd.user.services.niri-flake-polkit.enable = false` is intentional — DMS provides polkit itself; double-enabling causes conflicts.

### Power on Ryzen 7040

`power-profiles-daemon` is enabled; `tlp` is **explicitly disabled**. This is Framework's recommendation for Ryzen 7040 — TLP misbehaves on this platform. Don't swap them without a reason.

## Editing patterns

- **Adding a system-wide package**: append to `environment.systemPackages` in `configuration.nix`.
- **Adding a user CLI tool**: append to `home.packages` in `home.nix`. Prefer this over system packages unless the tool needs to be on PATH for other users or services.
- **Adding an imperative install step** (something not in nixpkgs): add a `home.activation.<name>` block in `home.nix` following the existing `cyberchef` / `lazyvimStarter` patterns — guard with an existence check so reruns are idempotent.
- **Bumping niri or DMS independently of nixpkgs**: `nix flake update niri` (or `dms`). The `--update-input` form on `nix flake lock` is deprecated since Nix 2.19.
- **`allowUnfree` is on** (`nixpkgs.config.allowUnfree = true`) for `lmstudio`, `typora`, `1password`. Unfree additions are fine.

## Secure Boot

Lanzaboote is a deliberate **post-install** step, not part of the initial build. The `lanzaboote` flake input and its module line in `flake.nix`, plus the `boot.lanzaboote` block in `configuration.nix`, are all commented out. Enabling them before keys are enrolled in the firmware will brick the boot. The full runbook is in `secure-boot.md`; touch those commented blocks only when following it.

When enabling, `lib` must be added to `configuration.nix`'s function arguments — currently it's `{ config, pkgs, inputs, ... }` because nothing uses `lib`. The lanzaboote block uses `lib.mkForce` to override systemd-boot.

## What's *not* declarative (by design)

Listed in `home.activation.todoMd` (the generated `~/TODO.md`): authenticating Claude Code, signing into 1Password / Gmail / GitHub / Slack / Discord / Signal / Zoom, setting wallpaper (DMS derives Material You theme from it), Obsidian Sync, Typora license, Chromium extensions, pulling Ollama models, `sudo fwupdmgr update`. These are credentials, account state, and firmware updates — not something Nix should own.
